--------------------------------------------------
FEATURE 1 — EXPERIMENT TEMPLATES
--------------------------------------------------

GOAL:
Allow users to create reusable experiment templates for recurring Synergy H1 workflows.

USE CASE:
A scientist repeatedly runs viability assays or ELISA assays with similar metadata and expected file structure.
Instead of filling everything from scratch, they can start from a template.

BACKEND:
Create a new SQLite table `experiment_templates` with fields:
- id
- name
- assay_type
- instrument (default “Synergy H1”)
- description
- default_notes
- expected_columns_json
- expected_control_rule
- expected_status_default
- ai_prompt_hint
- created_at
- updated_at

Meaning of fields:
- expected_columns_json: JSON array like ["Well", "Condition", "Dose_uM", "Signal"]
- expected_control_rule: plain text like “Condition = Control or Dose_uM = 0”
- ai_prompt_hint: extra context for AI like “This template is for cell viability dose-response assays”

API endpoints:
- GET /api/templates
- GET /api/templates/<id>
- POST /api/templates
- PUT /api/templates/<id>
- DELETE /api/templates/<id>

FRONTEND:
Add a “Templates” section or modal with:
- list of templates
- create template form
- edit template form
- delete template action

In the “New Experiment” flow:
- Add a “Start from template” dropdown
- When a template is selected, auto-fill:
  - assay_type
  - instrument
  - notes
  - optional expected parser hints
- Still allow the user to edit all fields manually

UI:
- Template cards should show:
  - template name
  - assay type
  - short description
  - instrument chip

--------------------------------------------------
FEATURE 2 — APPROVAL WORKFLOW FOR AI RECOMMENDATIONS
--------------------------------------------------

GOAL:
The AI can suggest next experiments, but a scientist or team lead must approve, reject, or edit them.

WHY:
This makes the app more realistic and trustworthy for lab workflows.

BACKEND:
Create a new SQLite table `recommendation_actions` with:
- id
- experiment_id
- recommendation_index
- recommendation_title
- original_recommendation_json
- edited_recommendation_json
- action_status (approved / rejected / edited / pending)
- reviewer_name
- reviewer_note
- created_at
- updated_at

If recommendations are already stored in ai_next_experiments_json on the experiment, keep using that.
This new table should track actions taken on those suggestions.

API endpoints:
- GET /api/experiments/<id>/recommendations/actions
- POST /api/experiments/<id>/recommendations/<index>/approve
- POST /api/experiments/<id>/recommendations/<index>/reject
- POST /api/experiments/<id>/recommendations/<index>/edit

For edit:
- accept updated title, variable_to_change, rationale, expected_outcome, confidence
- save edited_recommendation_json
- set action_status = edited

FRONTEND:
In the “Next Experiments” panel:
- For each recommendation card, add buttons:
  - Approve
  - Reject
  - Edit
- Add optional fields:
  - reviewer name
  - reviewer note
- If approved/rejected/edited, show a visible status chip on the card

UX:
- If recommendation is pending, show neutral color
- Approved = green
- Rejected = red
- Edited = amber/yellow
- Keep the original AI recommendation visible even if edited, with an expandable section “View original recommendation”

FEATURE 4 — TASK GENERATION FROM AI RECOMMENDATIONS
--------------------------------------------------

GOAL:
Allow the user to convert an approved AI recommendation into a tracked task.

USE CASE:
AI suggests a next experiment → scientist approves it → turns it into an actionable task.

BACKEND:
Create a new SQLite table `tasks` with:
- id
- experiment_id
- source_recommendation_index
- title
- description
- owner_name
- due_date
- status (todo / in_progress / done / blocked)
- priority (low / medium / high)
- created_at
- updated_at

API endpoints:
- GET /api/tasks
- GET /api/experiments/<id>/tasks
- POST /api/tasks
- PUT /api/tasks/<id>
- DELETE /api/tasks/<id> (optional)

Frontend:
For each approved recommendation:
- Add button “Create Task”
- Clicking it opens a small modal or inline form:
  - title (default from recommendation)
  - description (autofill from rationale + expected outcome)
  - owner name
  - due date
  - priority
- Save task and show success message

Add a new UI section:
- “Tasks” page or panel with:
  - filters by status and owner
  - grouped list or simple Kanban-like columns:
    - Todo
    - In Progress
    - Done
    - Blocked

Keep this relatively simple but polished.
