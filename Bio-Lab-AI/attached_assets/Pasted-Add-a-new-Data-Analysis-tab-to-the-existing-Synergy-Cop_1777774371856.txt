Add a new “Data Analysis” tab to the existing Synergy Copilot app.

GOAL:
Create a second main tab next to “Dashboard” called **“Data Analysis”**.

This tab should:
1) let the user choose an experiment (especially Synergy H1 / Gen5 plates),
2) show quantified metrics for that experiment, and
3) generate an AI-written “Data Analysis Report” using the existing copilot (Gemini).

ASSUMPTIONS:
- Experiments are already stored in SQLite.
- Synergy H1 / Gen5 CSV has been parsed into a structured summary (plate_summary_json or similar).
- There is already an /api/experiments endpoint and an /api/chat or AI endpoint.
- Frontend is React, backend is Flask, GEMINI_API_KEY is in Replit Secrets.

FRONTEND CHANGES:

1. Navigation:
- In the main nav (top or left), add a new tab or menu item: “Data Analysis”.
- When clicked, render a DataAnalysisPage component.

2. DataAnalysisPage layout:
- Top-left: Experiment selector (dropdown or list) showing experiments (name, date, assay_type, status).
- Top-right: Short experiment metadata card:
  - name, date, instrument (Synergy H1), assay type, status.
- Middle: “Quantitative Summary” panel:
  - Display parsed metrics from plate_summary_json:
    - Control mean, SD, CV%
    - Per-dose stats for Synergy H1:
      - dose_uM, mean_signal, SD, n, percent_change_vs_control
    - Outlier wells list
- Bottom: “AI Analysis Report” panel:
  - Button: “Generate AI Analysis”
  - Text area / read-only panel where the copilot’s analysis is shown.

3. UX details:
- When user selects an experiment, automatically load its quantitative summary (no AI yet).
- Only enable “Generate AI Analysis” if an experiment is selected and parsed metrics exist.
- Show loading state while AI is generating.
- If AI fails, show a clear error message.

BACKEND CHANGES:

1. Add or extend endpoint to fetch a single experiment with parsed summary:
- GET /api/experiments/<id>
  - Must include plate_summary_json (with stats as described below).

2. Add a dedicated AI endpoint for data analysis (if /api/chat is too generic):
- POST /api/analyze_experiment
  - Input JSON:
    - experiment_id
  - Backend logic:
    - Fetch experiment by id.
    - Fetch its plate_summary_json.
    - Fetch a few related experiments (same assay_type and instrument, if available).
    - Build a structured prompt to Gemini.
    - Return:
      - full_text_report (string)
      - optionally a short_summary and key_findings list.

EXPERIMENT SUMMARY FORMAT (plate_summary_json):

Assume plate_summary_json has fields like:

{
  "instrument": "Synergy H1",
  "assay_type": "viability",
  "control_stats": {
    "mean_signal": 0.985,
    "sd_signal": 0.050,
    "cv_percent": 5.08,
    "n_wells": 4
  },
  "dose_groups": [
    {
      "dose_uM": 0.1,
      "mean_signal": 0.805,
      "sd_signal": 0.021,
      "n_wells": 4,
      "percent_change_vs_control": -18.3
    },
    {
      "dose_uM": 1.0,
      "mean_signal": 0.275,
      "sd_signal": 0.035,
      "n_wells": 4,
      "percent_change_vs_control": -72.1
    }
  ],
  "outlier_wells": ["C3", "D4"]
}

If your summary format differs, adapt accordingly but keep it structured and numeric.

AI PROMPT DESIGN:

Use a new server-side system prompt tuned for data analysis:

SYSTEM PROMPT:

“You are an expert cell biology and lab data analysis copilot. You specialize in interpreting Synergy H1 / Gen5 microplate reader experiments. You receive:
1) a structured quantitative summary of the current experiment (control stats, dose-response stats, outliers),
2) metadata (assay type, instrument, notes),
3) optionally summaries of previous related experiments.

Your job is to:
- summarize what happened in this experiment,
- interpret the control and replicate quality,
- analyze the dose-response (or other pattern) in quantitative terms,
- identify likely causes of any failures or unexpected patterns,
- compare with previous experiments when relevant,
- and recommend 3 concrete next experiments to run.

Always:
- cite numeric data explicitly (e.g., “control mean 0.985, CV 5.1%”, “1.0 µM caused ~72% drop vs control”),
- distinguish between strong evidence and speculation,
- and keep recommendations practical for a small biotech lab.”

REQUEST CONSTRUCTION:

For POST /api/analyze_experiment:
- Use the system prompt above.
- Provide in the user/context messages:
  - experiment metadata (name, date, assay_type, instrument, notes).
  - plate_summary_json values (control_stats, dose_groups, outliers).
  - short summaries of up to 2–3 related experiments (if you have them).
- The user-facing ‘question’ can be something like:
  - “Please analyze this experiment’s data and produce a structured report.”

Ask Gemini to respond in structured markdown with sections:

1. Summary of Experiment
2. Data Quality & Controls
3. Dose-Response or Pattern Analysis
4. Likely Causes for Observed Outcomes
5. Comparison to Previous Experiments (if any)
6. Recommended Next 3 Experiments
7. Confidence and Limitations

FRONTEND DISPLAY:

In the “AI Analysis Report” panel:
- Render the markdown (or plain text with headings) from full_text_report.
- Provide a “Regenerate Analysis” button to re-run the endpoint if needed.

ERROR HANDLING:

- If plate_summary_json is missing or cannot be parsed, show a message:
  “No quantitative summary available for this experiment. Please upload a valid Synergy H1 / Gen5 CSV first.”
- If the AI call fails (missing GEMINI_API_KEY or network error), show:
  “The AI analysis could not be generated. Please check your API configuration or try again.”

ACCEPTANCE CRITERIA:

The “Data Analysis” tab is complete when:
1. I can open the tab and see a list of experiments.
2. I can select a Synergy H1 experiment and see numeric summary metrics from plate_summary_json.
3. I can click “Generate AI Analysis” and get a structured report based on that data.
4. The report clearly references actual numbers from the Synergy H1 summary.
5. Errors (no API key, no summary, bad experiment id) are handled with clear messages.
